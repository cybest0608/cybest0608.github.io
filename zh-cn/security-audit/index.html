MiraiChat 安全审计报告（专业版）

Language / 言語 / 语言 / 語言: English · 日本語 · 简体中文 · 繁體中文

日期： 2026-04-10
评估类型： 只读实现审计（后端 + 客户端）
适用对象： 安全工程师、技术产品负责人、基础设施运营人员

1. 执行摘要

MiraiChat 实现了一套可信的客户端加密消息架构，具备现代密码原语（X25519、Ed25519、AES-GCM、HMAC-SHA256）、安全的移动端密钥存储、证书固定逻辑、WebSocket 票据机制以及逐消息的临时认证。其基本安全态势明显优于传统的明文聊天系统，适合对隐私敏感的个人用户及中小企业使用。

然而，MiraiChat 在所有路径上并非最强 Signal 级别意义下的”服务器零可见”系统。后端仍会处理并存储大量元数据，将会话密钥存储在服务端用于验证，并支持推送/预览工作流，可能会暴露有限的可读上下文。安全结果同时取决于部署选择（TLS 终止、管理员加固、密钥管理和端点完整性）。

综合评级（实现层面）： 具备有界信任假设的良好实用隐私，而非敌对运营者假设下的最大保障保密性。

2. 范围与方法

  • 审查审计代码库中后端和客户端的当前实现行为。
  • 对加密、传输、认证/会话、存储、多设备流程及滥用面进行架构到实现的一致性检查。
  • 不做代码修改，不做渗透测试，不执行动态漏洞利用。

3. 威胁模型对齐

3.1 有力应对的威胁

  • 被动网络监听/MITM 尝试（通过 HTTPS/WSS + 固定控制 + 签名信任工件）。
  • 传输中消息的机会性篡改（AEAD 完整性检查）。
  • 出站消息的简单重放/伪造（nonce/时间戳/HMAC 临时令牌检查）。
  • 私钥向服务器运营者暴露（私钥保留在客户端）。

3.2 部分应对的威胁

  • 具有元数据访问权限的恶意/好奇的服务器运营者。
  • 通知面数据泄漏（预览导向数据和推送上下文）。
  • 多设备信任扩展（额外端点被攻陷会增大影响范围）。

3.3 设计上未应对（行业标准限制）

  • 端点/操作系统完全被攻陷、移动端恶意软件或硬件层信息窃取。
  • 超出 KDF 减速效果的用户弱认证熵。
  • 司法管辖区的审查/可用性管控（可用性问题，非密码破解）。

4. 密码学与密钥管理

4.1 原语与组合

  • 非对称密钥协商：X25519
  • 签名：Ed25519
  • 对称加密：AES-GCM-256
  • KDF 及令牌派生：PBKDF2-HMAC-SHA256 + HMAC-SHA256

这是适用于现代安全通信的行业合规原语集。

4.2 身份与密钥生命周期特性

  • 客户端身份/加密密钥材料通过 PBKDF2 从用户名 + PIN 确定性派生。
  • 运营优势：在相同的密钥输入下可恢复/可复现的密钥身份。
  • 安全注意事项：实际安全下限仍取决于 PIN 熵；KDF 仅增加暴力破解成本。

4.3 消息保护语义

  • 1:1 及群组消息的载荷加密在传输/路由前在客户端完成。
  • 棘轮/状态推进逻辑用于直接聊天流的前向保密行为。
  • 密封发件人信封路径降低了路由时服务器对发件方内部信息的可见性。

5. 传输与信道安全

5.1 HTTPS/WSS 态势

  • 正常运营以 API 的 HTTPS 和实时信道的 WSS 为基础构建。
  • WS 票据（短期有效）降低了套接字认证流中长期会话凭证的暴露风险。

5.2 证书信任控制

移动端固定路径已实现，包括：

  • 官方固定基线，
  • 自定义固定信任状态，
  • 签名固定列表导入，
  • 信任/审计事件跟踪，
  • 密钥轮换处理。

5.3 运营注意事项

实际传输保障依赖于部署配置：

  • 反向代理正确性，
  • 证书生命周期卫生，
  • 各环境的安全默认配置，
  • 源/主机策略兼容性。

6. 认证、会话与请求完整性

6.1 认证模型

  • 使用 bcrypt 的密码验证后端模型。
  • 登录时颁发会话密钥，并在服务端验证。

6.2 会话机制

  • 会话记录在服务端维护（包括设备上下文）。
  • 存在会话过期和清理逻辑。
  • WS 票据颁发和一次性验证加强了 WebSocket 准入控制。

6.3 逐消息的抗重放/真实性

  • 出站消息路径包含 nonce + 时间戳 + HMAC 令牌检查。
  • 使用重放缓存和时间戳窗口来降低重放的可行性。

6.4 管理面加固

  • 存在管理员基础认证、可选 TOTP 以及可选 IP 白名单控制。
  • 破坏性重置行为通过显式环境标志控制。

7. 服务端数据可见性与存储实况

7.1 存储内容

  • 账户：哈希凭证、身份/公开元数据。
  • 会话/设备：会话密钥、设备 ID/名称、推送令牌/平台。
  • 消息操作：离线队列载荷对象、未读/已读状态、投递元数据。
  • 文件：上传的二进制对象及元数据，以及令牌化的下载控制。

7.2 机密性边界（重要）

  • 主消息正文在正常加密路径中于客户端侧加密。
  • 服务器仍可见路由和状态所需的大量元数据。
  • 推送/预览功能可能暴露有限的内容上下文（短文本预览类），削弱严格机密性声明。

7.3 运营者可读性结论

  • 完整明文会话重建并非默认行为。
  • 部分上下文/元数据可见性依然真实存在且具有重要意义。
  • 因此，MiraiChat 应被定义为元数据可见的 E2EE，而非元数据最小化的匿名路由。

8. 多设备安全评估

8.1 关联流程态势

  • 设备关联交换使用临时密钥协商在设备间端对端加密。
  • 主设备授权和本地用户交互检查已存在。

8.2 风险影响

  • 关联设备的初始化将高度敏感的账户材料传输至辅助设备以引导其可用性。
  • 这在运营上是有效的，但提高了端点信任要求。
  • 安全保障与最薄弱的关联端点同等强度。

8.3 对 E2EE 保证的综合影响

  • 多设备支持本质上不会破坏 E2EE 传输语义。
  • 但确实扩大了受信任端点的边界和攻击面。

9. 滥用、误用与治理面

  • 用户屏蔽/举报工作流和本地黑名单行为已存在。
  • 强集中式审核/治理控制并非主要架构目标。
  • 平台行为更接近安全通信工具，而非重度审核的社交内容平台。

10. 对比定位（概念层面）

Signal

Signal 在严格最小化态势和反运营者机密性假设上仍更具优势。MiraiChat 更接近具有更大运营灵活性（包括自托管）的实用隐私方案。

WhatsApp

高级别的内容层 E2EE 概念相似，但两者均保留了重要的元数据面。MiraiChat 通过可选的自托管控制进行差异化，但相应地转移了运营者负担。

Telegram

MiraiChat 的默认加密聊天路径强于 Telegram 云聊天默认设置（E2EE 默认情况下并非普遍适用）。

11. 残余风险登记册（精简版）

高度相关

  • 端点被攻陷（移动端恶意软件/已 Root 设备/本地信息窃取）。
  • 尽管存在 PBKDF2 工作因子，用户 PIN 熵仍然较弱。
  • 自托管部署中 TLS 配置错误、固定信任工作流或管理员访问控制不当。

中度相关

  • 路由和推送层的元数据隐私泄漏。
  • 缺乏严格端点卫生策略下的多设备信任扩散。

低度相关（基于正常假设）

  • 原语级别的密码学弱点（未发现即时原语红色警报）。

12. 专业信任评估

对于以下威胁模型，合理谨慎的组织或高级用户可以信任 MiraiChat 用于私密通信：

  • 防止被动监听和常规服务端明文处理，
  • 对重放/篡改的适度抵御，
  • 受控的自托管或可信运营者环境。

MiraiChat 目前不适合以下威胁模型：

  • 接近零的元数据暴露，
  • 跨所有消息相关信道的严格反运营者机密性保证，
  • 高保证的端点攻陷抵御能力。

13. 部署与运营指南（无功能推测）

  • 将端点加固视为一级控制（MDM、操作系统补丁、越狱/Root 检测策略、应用锁态势）。
  • 实施强管理员控制（TOTP、严格白名单、密钥轮换规范）。
  • 运营稳健的 TLS 生命周期并监控信任/固定状态转换。
  • 根据监管和威胁要求对离线队列/文件保持保留最小化。
  • 记录面向用户的期望边界：加密无法抵御已被攻陷的设备。

14. 最终专业结论

MiraiChat 展示了具有实质性 E2EE 保护和现代信任控制的严肃且技术扎实的安全消息实现。在经过精心部署和运营的情况下，适用于许多实际的私密会话使用场景。

该系统应被诚实定位为具有明确运营和元数据注意事项的强实用隐私，而非针对所有特权基础设施行为者的绝对保密性。
