MiraiChat 后端自定义指南

Publish Date:   2026-03-27

MiraiChat 后端自定义指南

Language / 言語 / 语言 / 語言: English · 日本語 · 简体中文 · 繁體中文

这份指南面向有一定动手能力的用户：会用 Python、Docker，能做基础服务器配置，但不一定是专业后端工程师。

MiraiChat 的后端就是为这种“可渐进改造”场景设计的。

为什么 MiraiChat 支持自定义后端

MiraiChat 以隐私优先为核心，但每位用户的部署目标并不相同。

  • 有人希望数据完全自主管理
  • 有人需要按地区与网络环境做优化
  • 有人想接入自己的自动化、审核或内部系统

自定义后端的价值在于：在保持同一套应用体验的前提下，你可以自己决定

  • 数据存在哪里
  • 消息怎么路由
  • 安全与保留策略怎么定
  • 还要扩展哪些能力

后端可以自定义哪些内容

不需要一次改完。建议从一个模块开始。

1) 消息路由与投递策略

可调整：

  • 在线投递优先级
  • 跨节点 fanout 行为
  • 离线队列策略
  • 多设备自同步规则

收益：更好控制时延、可靠性与多设备体验。

2) 认证与会话策略

可调整：

  • 会话过期策略
  • 注册模式（开放或注册码）
  • 管理员保护规则
  • token 校验严格度

收益：可做更严格私有化安全，也可做更顺滑的可信网络接入。

3) WebSocket 事件处理链路

可调整：

  • 接受哪些事件类型
  • 事件校验逻辑
  • 各事件的路由策略
  • 回执/删除/系统事件处理方式

收益：可以为你的业务添加专用事件类型。

4) 推送通知策略

可调整：

  • 哪类消息触发推送
  • 同步/异步发送策略
  • 推送通道优先级与回退路径
  • hub/私服拓扑中的 relay 行为

收益：在电量、网络成本与触达率之间取得更合适的平衡。

5) 存储与保留策略

可调整：

  • 用户/媒体存储上限
  • 文件保留时间
  • 离线消息保留策略
  • 清理任务调度规则

收益：可从低保留隐私模式扩展到较大规模自托管归档。

6) 基础设施路径（单机到多节点）

可调整：

  • SQLite 起步部署
  • memory/Redis 的 presence 与 fanout 切换
  • 后续扩容迁移路径

收益：先快速上线，后续按需扩展。

自建后端能获得什么

  • 数据所有权：服务器、数据库、保留规则都由你控制
  • 本地化策略：按你的地区网络现实做优化
  • 快速迭代：改完就能立即验证
  • 集成自由：可接入自有 API、身份系统、自动化与审核能力
  • 不被固定厂商绑定：模型与流程可按你需要演进

WebSocket 直连模式 vs 隧道模式

MiraiChat 同时支持两种模式，因为现实网络条件差异很大。

直连模式 (WS_CONNECTION_MODE=direct)

适用场景：你能提供稳定公网入口。

  • 一般延迟更低
  • 路径更短、更可预测
  • 更适合追求极致性能

隧道模式 (WS_CONNECTION_MODE=tunnel)

适用场景：不方便直接暴露公网入站端口。

  • 适合严格 NAT 或家庭网络
  • 自托管远程可达性通常更容易
  • 相比直连通常会增加少量延迟

简单建议：重视最低延迟选 direct，重视部署便利选 tunnel。

建议的第一步改造路径

  1. 先用默认认证/会话策略跑通基线
  2. 增加一个小型 endpoint 或一个 WS 事件
  3. 调整一项保留策略并观察运行情况
  4. 在单节点瓶颈出现后再引入 Redis fanout

给实践者的鼓励

你不需要是后端专家，依然可以把 MiraiChat 后端改成真正适合自己的形态。

从一个小改动开始，反复测试与迭代，很快就能看到明显收益。
