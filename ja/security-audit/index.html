MiraiChat セキュリティ監査レポート（プロフェッショナル版）

Language / 言語 / 语言 / 語言: English · 日本語 · 简体中文 · 繁體中文

日付: 2026-04-10
評価種別: 読み取り専用実装監査（バックエンド＋クライアント）
対象読者: セキュリティエンジニア、テクニカルプロダクトオーナー、インフラ運用担当者

1. エグゼクティブサマリー

MiraiChat は、最新の暗号プリミティブ（X25519、Ed25519、AES-GCM、HMAC-SHA256）、セキュアなモバイルシークレットストレージ、証明書ピン留めロジック、WebSocket チケット発行、メッセージごとのエフェメラル認証を用いた、信頼性の高いクライアントサイド暗号化メッセージングアーキテクチャを実装しています。このベースラインのセキュリティ態勢は、従来のプレーンテキストチャットシステムと比較して大幅に優れており、プライバシー重視の一般ユーザーおよび SMB ユーザーの運用上の使用に適しています。

ただし、MiraiChat はすべてのパスにわたって Signal のような最も強い意味での「サーバー可視性ゼロ」システムではありません。バックエンドは依然として意味のあるメタデータを処理・保存し、検証のためにセッションシークレットをサーバーサイドで保存し、限定された人間が読めるコンテキストを公開する可能性のあるプッシュ/プレビューワークフローをサポートしています。セキュリティの結果は、デプロイメントの選択（TLS 終端、管理者のハードニング、鍵管理、エンドポイント整合性）にも大きく依存します。

総合評価（実装レベル）: 有界な信頼前提を持つ優れた実用的プライバシー。敵対的オペレーター前提下での最大保証の秘密性ではありません。

2. スコープと方法論

  • 監査対象コードベースに存在するバックエンドとクライアントの現在の実装動作をレビュー。
  • 暗号化、トランスポート、認証/セッション、ストレージ、マルチデバイスフロー、悪用サーフェスについて、アーキテクチャ対実装の整合性チェックを実施。
  • コード変更なし、ペネトレーションテストなし、動的なエクスプロイト実行なし。

3. 脅威モデルの整合性

3.1 強く対処されている脅威

  • パッシブなネットワーク傍受/MITM 試行（HTTPS/WSS + ピン留めコントロール + 署名付きトラストアーティファクトによる）。
  • 転送中のメッセージの日和見的な改ざん（AEAD 整合性チェック）。
  • 送信メッセージの単純なリプレイ/偽造（ノンス/タイムスタンプ/HMAC エフェメラルトークンチェック）。
  • サーバーオペレーターへの秘密鍵の露出（秘密鍵はクライアントサイドに保持）。

3.2 部分的に対処されている脅威

  • メタデータアクセスを持つ悪意のある/好奇心旺盛なサーバーオペレーター。
  • 通知プレーンの漏洩（プレビュー指向データとプッシュコンテキスト）。
  • マルチデバイスの信頼拡張（追加のエンドポイント侵害により被害範囲が拡大）。

3.3 設計上対処されていない（業界標準の制限）

  • エンドポイント/OS の完全な侵害、モバイルマルウェア、またはハードウェアの不正取得。
  • KDF 処理負荷の効果を超えたユーザー側の弱い認証エントロピー。
  • 管轄権に基づく検閲/可用性制御（可用性であり、暗号の破壊ではない）。

4. 暗号化と鍵管理

4.1 プリミティブと構成

  • 非対称鍵共有: X25519
  • 署名: Ed25519
  • 対称暗号化: AES-GCM-256
  • KDF とトークン派生: PBKDF2-HMAC-SHA256 + HMAC-SHA256

これは現代のセキュアメッセージングに適した業界準拠のプリミティブセットです。

4.2 ID と鍵ライフサイクルの特性

  • クライアントの ID/暗号化鍵材料は、PBKDF2 を介してユーザー名 + PIN 入力から決定論的に派生されます。
  • 運用上の利点: 同じシークレット入力の下で回復可能/再現可能な鍵 ID。
  • セキュリティ上の注意点: 実効的なセキュリティの下限は依然として PIN エントロピーに依存し、KDF はブルートフォースコストを増加させるだけです。

4.3 メッセージ保護のセマンティクス

  • 1:1 およびグループのペイロード暗号化は、トランスポート/ルーティング前にクライアントで発生します。
  • ラチェット/状態進行ロジックは、直接チャットフローにおける前方秘密性の動作のために存在します。
  • シールドセンダーエンベロープパスは、ルーティング中のサーバーが送信者側の内部を見ることを減らします。

5. トランスポートとチャネルセキュリティ

5.1 HTTPS/WSS の姿勢

  • 通常の運用は、API 用の HTTPS とリアルタイムチャネル用の WSS を中心に構築されています。
  • WS チケット（短命）は、ソケット認証フローにおける長命セッション認証情報の露出を減らします。

5.2 証明書トラストコントロール

モバイルのピン留めパスが実装されており、以下を含みます:

  • 公式のピンベースライン、
  • カスタムピントラスト状態、
  • 署名付きピンリストインポート、
  • トラスト/監査イベント追跡、
  • 鍵ローテーション処理。

5.3 運用上の注意点

実効的なトランスポートの保証はデプロイメントに依存します:

  • リバースプロキシの正確性、
  • 証明書ライフサイクルのヒジーン、
  • 環境ごとのセキュアなデフォルト、
  • オリジン/ホストポリシーの互換性。

6. 認証、セッション、リクエスト整合性

6.1 認証モデル

  • 保存時の bcrypt を使ったパスワード検証バックエンドモデル。
  • ログイン時にセッションシークレットが発行され、サーバーサイドで検証されます。

6.2 セッションメカニクス

  • セッションレコードはサーバーサイドで維持されます（デバイスコンテキストを含む）。
  • セッションの有効期限とクリーンアップロジックが存在します。
  • WS チケット発行と単一使用検証により、WebSocket への入場が強化されます。

6.3 メッセージごとのアンチリプレイ/真正性

  • 送信メッセージパスには、ノンス + タイムスタンプ + HMAC トークンチェックが含まれます。
  • リプレイキャッシュとタイムスタンプウィンドウが使用され、リプレイの実現可能性を低減します。

6.4 管理サーフェスのハードニング

  • 管理者基本認証、オプションの TOTP、およびオプションの IP 許可リストコントロールが存在します。
  • 破壊的なリセット動作は、明示的な環境フラグによってゲートされます。

7. サーバーサイドのデータ可視性とストレージの実態

7.1 保存されるもの

  • アカウント: ハッシュ化された認証情報、ID/パブリックメタデータ。
  • セッション/デバイス: セッションシークレット、デバイス ID/名前、プッシュトークン/プラットフォーム。
  • メッセージ操作: オフラインキューペイロードオブジェクト、未読/既読状態、配信メタデータ。
  • ファイル: アップロードされたバイナリオブジェクト、メタデータ、トークン化されたダウンロードコントロール。

7.2 機密性の境界（重要）

  • 主なメッセージ本文は、通常の暗号化パスではクライアントサイドで暗号化されています。
  • サーバーは依然としてルーティングと状態に必要な大量のメタデータを参照します。
  • プッシュ/プレビュー機能は、限定的なコンテキスト（短いテキストプレビュークラス）を公開する可能性があり、厳格な機密性の主張を減少させます。

7.3 オペレーターの可読性に関する結論

  • 完全なプレーンテキスト会話の再構築はデフォルトの動作ではありません。
  • 部分的なコンテキスト/メタデータの可視性は依然として実在し、重要です。
  • したがって、MiraiChat はメタデータ可視の E2EE としてモデル化されるべきであり、メタデータ最小化された匿名ルーティングではありません。

8. マルチデバイスセキュリティ評価

8.1 リンクフローの姿勢

  • デバイスリンク交換は、エフェメラル鍵合意を使用してデバイス間でエンドツーエンドで暗号化されます。
  • メインデバイスの認証とローカルユーザーインタラクションチェックが存在します。

8.2 リスクの影響

  • リンクされたデバイスのオンボーディングは、セカンダリデバイスの使用可能性をブートストラップするために非常に機密性の高いアカウント材料を転送します。
  • これは運用上は効果的ですが、エンドポイントの信頼要件を高めます。
  • セキュリティ保証は、最も弱いリンクされたエンドポイントと同じ強さです。

8.3 E2EE 保証への正味の影響

  • マルチデバイスサポートは本質的に E2EE トランスポートのセマンティクスを破壊しません。
  • ただし、信頼されたエンドポイントの境界と侵害サーフェスを拡大します。

9. 悪用、誤用、ガバナンスサーフェス

  • ユーザーブロック/報告ワークフローとローカルブロックリスト動作が存在します。
  • 強力な集中型モデレーション/ガバナンスコントロールは、主要なアーキテクチャ目標ではありません。
  • プラットフォームの動作は、重度のモデレーションが施されたソーシャルコンテンツプラットフォームよりも、セキュアな通信ツールに近いです。

10. 比較ポジショニング（概念的）

Signal

Signal は、厳格な最小化姿勢と反オペレーター機密性前提において依然として優れています。MiraiChat は、より大きな運用柔軟性（セルフホスティングを含む）を持つ実用的なプライバシーに近いです。

WhatsApp

高レベルの E2EE-at-content の概念は類似していますが、両者とも意味のあるメタデータプレーンを保持します。MiraiChat は、オプションのセルフホスティング制御で差別化しますが、それに応じてオペレーターの負担を移転します。

Telegram

MiraiChat のデフォルトの暗号化チャットパスは、Telegram のクラウドチャットのデフォルト（E2EE がデフォルトで普遍的ではない）よりも強力です。

11. 残余リスク登録簿（要約）

高い関連性

  • エンドポイント侵害（モバイルマルウェア/ルート化デバイス/ローカル不正取得）。
  • PBKDF2 ワークファクターにもかかわらず弱いユーザー PIN エントロピー。
  • セルフホスト型デプロイメントにおける誤設定された TLS、ピントラストワークフロー、または管理者アクセスコントロール。

中程度の関連性

  • ルーティングおよびプッシュレイヤーでのメタデータプライバシー漏洩。
  • 厳格なエンドポイントヒジーンポリシーなしのマルチデバイスの信頼の拡散。

低い関連性（通常の前提を考慮）

  • プリミティブレベルの暗号学的弱点（観察された即時のプリミティブのレッドフラグなし）。

12. プロフェッショナルな信頼評価

脅威モデルが以下の場合、合理的に慎重な組織または上級ユーザーは MiraiChat をプライベート通信として信頼できます:

  • パッシブな傍受と通常のサーバーサイドのプレーンテキスト処理からの保護、
  • リプレイ/改ざんへの適度な耐性、
  • 制御されたセルフホストまたは信頼されたオペレーター環境。

MiraiChat は現状では以下の脅威モデルには適していません:

  • ほぼゼロのメタデータ露出、
  • すべてのメッセージ関連チャネルにわたる厳格な反オペレーター機密性保証、
  • 高保証のエンドポイント侵害耐性。

13. デプロイメントと運用ガイダンス（機能推測なし）

  • エンドポイントハードニングを第一次コントロールとして扱う（MDM、OS パッチ適用、ジェイルブレイク/ルート検知ポリシー、アプリロック姿勢）。
  • 強力な管理者コントロールを実施する（TOTP、厳格な許可リスト、シークレットローテーション規律）。
  • 堅牢な TLS ライフサイクルを運用し、トラスト/ピン状態の遷移を監視する。
  • 規制および脅威要件に一致したオフラインキュー/ファイルの保持最小化を維持する。
  • ユーザー向けの期待値の境界を文書化する: 暗号化は侵害されたデバイスを破ることはできない。

14. 最終プロフェッショナル結論

MiraiChat は、意味のある E2EE 保護と最新のトラストコントロールを備えた、真剣で技術的に根拠のあるセキュアメッセージング実装を示しています。慎重にデプロイおよび運用された場合、多くの実用的なプライベート会話ユースケースに適しています。

このシステムは、すべての特権インフラアクターに対する絶対的な秘密性としてではなく、明示的な運用上およびメタデータ上の注意事項を持つ強力な実用的プライバシーとして誠実にポジショニングされるべきです。
