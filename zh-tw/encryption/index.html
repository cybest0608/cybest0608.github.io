MiraiChat 密碼學技術參考

Language / 言語 / 语言 / 語言: English · 日本語 · 简体中文 · 繁體中文

範圍

本文檔總結了當前在用戶端代碼庫中實現的加密設計。

1. 核心原語

  • 非對稱金鑰協議：X25519
  • 非對稱簽名：Ed25519
  • 對稱 AEAD：AES-GCM-256
  • PIN KDF：PBKDF2(HMAC-SHA256, iterations=600000, bits=256)
  • 棘輪 KDF 元件：HMAC-SHA256

相關實現：

  • lib/key_manager.dart

2. 身分與裝置金鑰材料

KeyManager.init(username, pin) 使用 PBKDF2 從使用者 PIN + 使用者名稱鹽衍生出一個確定性的基礎種子。然後將兩個確定性種子拆分為：

  • 加密金鑰對種子 (X25519)
  • 身分簽名金鑰對種子 (Ed25519)

影響：

  • 金鑰對可以從相同的使用者名稱/PIN 輸入中重現。
  • 安全強度與 PIN 熵 + PBKDF2 工作因子掛鉤。

3. 1:1 訊息加密流程

對於直接訊息：

  1. 用戶端使用對等公鑰 (X25519) 衍生共享金鑰。
  2. 主路徑（主裝置）：衍生並推進每條訊息的棘輪金鑰。
  3. 相容路徑（輔助/非棘輪模式）：使用穩定的共享金鑰。
  4. 訊息負載被加密為 enc:<base64-aesgcm-box>。

棘輪行為：

  • 棘輪狀態：每個對話包含 ratchet + counter。
  • 訊息金鑰和下一個棘輪狀態通過 HMAC-SHA256 域分離標籤衍生。
  • 狀態在推進後持久化到安全存儲。

相關实现：

  • lib/outgoing_message_queue.dart
  • lib/incoming_message_handler.dart
  • lib/key_manager.dart

4. 群組訊息加密

  • 群組金鑰是對稱的，通過 AES-GCM 金鑰產生器生成。
  • 群組內容在群組金鑰下加密 (AES-GCM-256)。
  • 成員變更時可以觸發群組金鑰刷新/輪換。
  • 分發給成員時使用每成員共享金鑰加密進行包裝。

相關实现：

  • lib/incoming_message_handler.dart (_rotateGroupKeyAfterMembershipChange)
  • lib/key_manager.dart (generateGroupKey)

5. 密封發送者路徑 (Sealed Sender)

外發的 1:1 訊息可能被封裝在一個密封信封中，包含：

  • sealed_payload = 加密的內部信封
  • 來自伺服器的 sender_certificate
  • protocol_version = 1

接收方驗證：

  1. 憑證格式和簽名 (Ed25519) 使用伺服器簽名金鑰驗證。
  2. 憑證有效期。
  3. 使用共享金鑰解密密封負載。

相關实现：

  • lib/outgoing_message_queue.dart (密封發送者構建)
  • lib/incoming_message_handler.dart (_handleSealedSender)
  • lib/certificate_pinning_manager.dart (伺服器簽名金鑰 + 固定列表驗證)

6. 傳輸與信任模型

  • HTTPS/WSS 傳輸用於 API/WebSocket 通訊。
  • 憑證釘選管理器支援：
    • 硬編碼的官方釘選 (pins)
    • 有界的自定義釘選信任歷史
    • 帶有 Ed25519 驗證的簽名釘選列表匯入
  • 信任稽核事件和輪換中繼資料記錄在偏好設定中。

相關实现：

  • lib/chat_api.dart
  • lib/websocket_connection_manager.dart
  • lib/certificate_pinning_manager.dart

7. 本地機密存儲

機密通過平台安全存儲：

  • iOS Keychain
  • Android 加密安全存儲

存儲的敏感項包括工作階段機密、憑據機密、復原種子和棘輪狀態。

相關实现：

  • lib/secure_storage.dart

8. 重要當前安全注意事項

  • LocalMessageStore.decryptContent 目前直接返回輸入；本地訊息資料庫保護不由此函數進行全內容加密 (lib/local_message_store.dart)。
  • 使用者名稱 API 哈希使用多輪 SHA-256 和全域常量鹽；這是查找協議的混淆/衍生行為，不是密碼 KDF 的替代。
  • 從 PIN 確定性生成金鑰在操作上很方便，但為了獲得最佳安全性，需要強大的 PIN 策略。

9. 威脅覆蓋（當前）

涵蓋：

  • 被動網路攔截 (TLS + E2EE 負載)
  • 伺服器憑證不匹配/篡改檢測 (釘選)
  • 發送者憑證篡改 (簽名驗證)
  • 傳輸中的訊息機密性/完整性 (AES-GCM)

部分涵蓋 / 取決於設計：

  • 裝置受損（取決於作業系統加固和應用鎖定策略）
  • 弱使用者 PIN 選擇（PBKDF2 減慢暴力破解但無法修復低熵）
  • 中繼資料隱私（路由中繼資料按協議必要性仍然存在）
