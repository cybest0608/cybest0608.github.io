MiraiChat 安全稽核報告（專業版）

Language / 言語 / 语言 / 語言: English · 日本語 · 简体中文 · 繁體中文

日期： 2026-04-10
評估類型： 唯讀實作稽核（後端 + 客戶端）
適用對象： 安全工程師、技術產品負責人、基礎設施運營人員

1. 執行摘要

MiraiChat 實現了一套可信的客戶端加密訊息架構，具備現代密碼原語（X25519、Ed25519、AES-GCM、HMAC-SHA256）、安全的行動端金鑰儲存、憑證固定邏輯、WebSocket 票證機制以及逐訊息的臨時認證。其基本安全態勢明顯優於傳統的明文聊天系統，適合對隱私敏感的個人用戶及中小企業使用。

然而，MiraiChat 在所有路徑上並非最強 Signal 級別意義下的「伺服器零可見」系統。後端仍會處理並儲存大量元資料，將工作階段金鑰儲存於伺服器端用於驗證，並支援推播/預覽工作流程，可能會暴露有限的可讀上下文。安全結果同時取決於部署選擇（TLS 終止、管理員加固、金鑰管理和端點完整性）。

總體評級（實作層面）： 具備有界信任假設的良好實用隱私，而非敵對運營者假設下的最大保障保密性。

2. 範圍與方法

  • 審查稽核程式庫中後端和客戶端的當前實作行為。
  • 對加密、傳輸、認證/工作階段、儲存、多裝置流程及濫用面進行架構到實作的一致性檢查。
  • 不做程式碼修改，不做滲透測試，不執行動態漏洞利用。

3. 威脅模型對齊

3.1 有力應對的威脅

  • 被動網路監聽/MITM 嘗試（透過 HTTPS/WSS + 固定控制 + 簽名信任工件）。
  • 傳輸中訊息的機會性篡改（AEAD 完整性檢查）。
  • 出站訊息的簡單重放/偽造（nonce/時間戳記/HMAC 臨時令牌檢查）。
  • 私鑰向伺服器運營者暴露（私鑰保留在客戶端）。

3.2 部分應對的威脅

  • 具有元資料存取權限的惡意/好奇的伺服器運營者。
  • 通知面資料洩漏（預覽導向資料和推播上下文）。
  • 多裝置信任擴展（額外端點被攻陷會增大影響範圍）。

3.3 設計上未應對（行業標準限制）

  • 端點/作業系統完全被攻陷、行動端惡意軟體或硬體層資訊竊取。
  • 超出 KDF 減速效果的使用者弱認證熵。
  • 司法管轄區的審查/可用性管控（可用性問題，非密碼破解）。

4. 密碼學與金鑰管理

4.1 原語與組合

  • 非對稱金鑰協商：X25519
  • 簽名：Ed25519
  • 對稱加密：AES-GCM-256
  • KDF 及令牌派生：PBKDF2-HMAC-SHA256 + HMAC-SHA256

這是適用於現代安全通訊的行業合規原語集。

4.2 身份與金鑰生命週期特性

  • 客戶端身份/加密金鑰材料透過 PBKDF2 從使用者名稱 + PIN 確定性派生。
  • 運營優勢：在相同金鑰輸入下可恢復/可復現的金鑰身份。
  • 安全注意事項：實際安全下限仍取決於 PIN 熵；KDF 僅增加暴力破解成本。

4.3 訊息保護語義

  • 1:1 及群組訊息的載荷加密在傳輸/路由前在客戶端完成。
  • 棘輪/狀態推進邏輯用於直接聊天流的前向保密行為。
  • 密封發件人訊封路徑降低了路由時伺服器對發件方內部資訊的可見性。

5. 傳輸與頻道安全

5.1 HTTPS/WSS 態勢

  • 正常運營以 API 的 HTTPS 和即時頻道的 WSS 為基礎建構。
  • WS 票證（短期有效）降低了套接字認證流中長期工作階段憑證的暴露風險。

5.2 憑證信任控制

行動端固定路徑已實作，包括：

  • 官方固定基準，
  • 自訂固定信任狀態，
  • 簽名固定清單匯入，
  • 信任/稽核事件追蹤，
  • 金鑰輪換處理。

5.3 運營注意事項

實際傳輸保障依賴於部署配置：

  • 反向代理正確性，
  • 憑證生命週期衛生，
  • 各環境的安全預設配置，
  • 來源/主機策略相容性。

6. 認證、工作階段與請求完整性

6.1 認證模型

  • 使用 bcrypt 的密碼驗證後端模型。
  • 登入時頒發工作階段金鑰，並在伺服器端驗證。

6.2 工作階段機制

  • 工作階段記錄在伺服器端維護（包括裝置上下文）。
  • 存在工作階段過期和清理邏輯。
  • WS 票證頒發和一次性驗證強化了 WebSocket 準入控制。

6.3 逐訊息的抗重放/真實性

  • 出站訊息路徑包含 nonce + 時間戳記 + HMAC 令牌檢查。
  • 使用重放快取和時間戳記視窗來降低重放的可行性。

6.4 管理面加固

  • 存在管理員基礎認證、可選 TOTP 以及可選 IP 白名單控制。
  • 破壞性重置行為透過顯式環境旗標控制。

7. 伺服器端資料可見性與儲存實況

7.1 儲存內容

  • 帳戶：雜湊憑證、身份/公開元資料。
  • 工作階段/裝置：工作階段金鑰、裝置 ID/名稱、推播令牌/平台。
  • 訊息操作：離線佇列載荷物件、未讀/已讀狀態、投遞元資料。
  • 檔案：上傳的二進位物件及元資料，以及令牌化的下載控制。

7.2 機密性邊界（重要）

  • 主要訊息正文在正常加密路徑中於客戶端側加密。
  • 伺服器仍可見路由和狀態所需的大量元資料。
  • 推播/預覽功能可能暴露有限的內容上下文（短文字預覽類），削弱嚴格機密性聲明。

7.3 運營者可讀性結論

  • 完整明文對話重建並非預設行為。
  • 部分上下文/元資料可見性依然真實存在且具有重要意義。
  • 因此，MiraiChat 應被定義為元資料可見的 E2EE，而非元資料最小化的匿名路由。

8. 多裝置安全評估

8.1 關聯流程態勢

  • 裝置關聯交換使用臨時金鑰協商在裝置間端到端加密。
  • 主裝置授權和本地使用者互動檢查已存在。

8.2 風險影響

  • 關聯裝置的初始化將高度敏感的帳戶材料傳輸至次要裝置以引導其可用性。
  • 這在運營上是有效的，但提高了端點信任要求。
  • 安全保障與最薄弱的關聯端點同等強度。

8.3 對 E2EE 保證的綜合影響

  • 多裝置支援本質上不會破壞 E2EE 傳輸語義。
  • 但確實擴大了受信任端點的邊界和攻擊面。

9. 濫用、誤用與治理面

  • 使用者封鎖/檢舉工作流程和本地封鎖清單行為已存在。
  • 強集中式審核/治理控制並非主要架構目標。
  • 平台行為更接近安全通訊工具，而非重度審核的社交內容平台。

10. 對比定位（概念層面）

Signal

Signal 在嚴格最小化態勢和反運營者機密性假設上仍更具優勢。MiraiChat 更接近具有更大運營彈性（包括自架服務）的實用隱私方案。

WhatsApp

高級別的內容層 E2EE 概念相似，但兩者均保留了重要的元資料面。MiraiChat 透過可選的自架服務控制進行差異化，但相應地轉移了運營者負擔。

Telegram

MiraiChat 的預設加密聊天路徑強於 Telegram 雲端聊天預設設定（E2EE 預設情況下並非普遍適用）。

11. 殘余風險登記冊（精簡版）

高度相關

  • 端點被攻陷（行動端惡意軟體/已 Root 裝置/本地資訊竊取）。
  • 儘管存在 PBKDF2 工作因子，使用者 PIN 熵仍然較弱。
  • 自架服務部署中 TLS 配置錯誤、固定信任工作流程或管理員存取控制不當。

中度相關

  • 路由和推播層的元資料隱私洩漏。
  • 缺乏嚴格端點衛生策略下的多裝置信任擴散。

低度相關（基於正常假設）

  • 原語級別的密碼學弱點（未發現即時原語紅色警報）。

12. 專業信任評估

對於以下威脅模型，合理謹慎的組織或進階使用者可以信任 MiraiChat 用於私密通訊：

  • 防止被動監聽和常規伺服器端明文處理，
  • 對重放/篡改的適度抵禦，
  • 受控的自架服務或可信運營者環境。

MiraiChat 目前不適合以下威脅模型：

  • 接近零的元資料暴露，
  • 跨所有訊息相關頻道的嚴格反運營者機密性保證，
  • 高保證的端點攻陷抵禦能力。

13. 部署與運營指南（無功能推測）

  • 將端點加固視為一級控制（MDM、作業系統修補、越獄/Root 偵測策略、應用程式鎖態勢）。
  • 實施強管理員控制（TOTP、嚴格白名單、金鑰輪換規範）。
  • 運營穩健的 TLS 生命週期並監控信任/固定狀態轉換。
  • 根據法規和威脅要求對離線佇列/檔案保持保留最小化。
  • 記錄面向使用者的期望邊界：加密無法抵禦已被攻陷的裝置。

14. 最終專業結論

MiraiChat 展示了具有實質性 E2EE 保護和現代信任控制的嚴肅且技術扎實的安全訊息實作。在經過精心部署和運營的情況下，適用於許多實際的私密對話使用場景。

該系統應被誠實定位為具有明確運營和元資料注意事項的強實用隱私，而非針對所有特權基礎設施行為者的絕對保密性。
