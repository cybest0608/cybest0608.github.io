MiraiChat Security Audit Report (Professional Edition)

Publish Date:   2026-04-10

MiraiChat Security Audit Report (Professional Edition)

Language / 言語 / 语言 / 語言: English · 日本語 · 简体中文 · 繁體中文

Date: 2026-04-10
Assessment type: Read-only implementation audit (backend + client)
Audience: Security engineers, technical product owners, infrastructure operators

1. Executive Summary

MiraiChat implements a credible client-side encrypted messaging architecture with modern primitives (X25519, Ed25519, AES-GCM, HMAC-SHA256), secure mobile secret storage, certificate pinning logic, WebSocket ticketing, and per-message ephemeral authentication. The baseline posture is materially stronger than conventional plaintext chat systems and operationally suitable for privacy-sensitive consumer and SMB usage.

However, MiraiChat is not a strict “zero server visibility” system in the strongest Signal-like sense across all paths. The backend still processes and stores meaningful metadata, stores session secrets server-side for validation, and supports push/preview workflows that can expose limited human-readable context. Security outcomes also depend significantly on deployment choices (TLS termination, admin hardening, key management, and endpoint integrity).

Overall rating (implementation-level): Good practical privacy with bounded trust assumptions, not maximum-assurance secrecy under hostile-operator assumptions.

2. Scope and Method

  • Reviewed current backend and client implementation behavior as present in the audited codebase.
  • Performed architecture-to-implementation consistency checks for encryption, transport, auth/session, storage, multi-device flows, and abuse surfaces.
  • No code changes, no penetration testing, no dynamic exploit execution.

3. Threat Model Alignment

3.1 Strongly addressed threats

  • Passive network interception/MITM attempts (via HTTPS/WSS + pinning controls + signed trust artifacts).
  • Opportunistic message tampering in transit (AEAD integrity checks).
  • Simple replay/forgery of outbound messages (nonce/timestamp/HMAC ephemeral token checks).
  • Exposure of private keys to server operators (private keys remain client-side).

3.2 Partially addressed threats

  • Malicious/curious server operator with metadata access.
  • Notification-plane leakage (preview-oriented data and push context).
  • Multi-device trust expansion (additional endpoint compromises increase blast radius).

3.3 Not addressed by design (industry-standard limitation)

  • Fully compromised endpoint/OS, mobile malware, or hardware exfiltration.
  • User-side weak authentication entropy beyond KDF slowing effects.
  • Jurisdictional censorship/availability controls (availability, not cipher break).

4. Cryptography and Key Management

4.1 Primitives and composition

  • Asymmetric agreement: X25519
  • Signatures: Ed25519
  • Symmetric encryption: AES-GCM-256
  • KDF and token derivations: PBKDF2-HMAC-SHA256 + HMAC-SHA256

This is an industry-appropriate primitive set for modern secure messaging.

4.2 Identity and key lifecycle characteristics

  • Client identity/encryption key material is deterministically derived from username + PIN input via PBKDF2.
  • Operational benefit: recoverable/reproducible key identity under the same secret inputs.
  • Security caveat: effective security floor still depends on PIN entropy; KDF only increases brute-force cost.

4.3 Message protection semantics

  • 1:1 and group payload encryption occurs on client before transport/routing.
  • Ratchet/state advancement logic exists for forward secrecy behavior in direct chat flows.
  • Sealed-sender envelope path reduces server visibility of sender-side internals during routing.

5. Transport and Channel Security

5.1 HTTPS/WSS posture

  • Normal operation is built around HTTPS for API and WSS for realtime channels.
  • WS ticketing (short-lived) reduces exposure of long-lived session credentials in socket auth flows.

5.2 Certificate trust controls

Mobile pinning path is implemented, including:

  • official pin baselines,
  • custom pin trust state,
  • signed pin-list import,
  • trust/audit event tracking,
  • key-rotation handling.

5.3 Operational caveat

Effective transport assurances remain deployment-dependent:

  • reverse proxy correctness,
  • certificate lifecycle hygiene,
  • secure defaults per environment,
  • origin/host policy compatibility.

6. Authentication, Session, and Request Integrity

6.1 Authentication model

  • Password-verification backend model with bcrypt at rest.
  • Session-secret issued at login and validated server-side.

6.2 Session mechanics

  • Session records are maintained server-side (including device context).
  • Session expiry and cleanup logic is present.
  • WS ticket issuance and single-use validation harden websocket admission.

6.3 Per-message anti-replay/authenticity

  • Outbound message paths include nonce + timestamp + HMAC token checks.
  • Replay cache and timestamp windows are used to reduce replay viability.

6.4 Admin-surface hardening

  • Admin basic auth, optional TOTP, and optional IP allowlist controls exist.
  • Destructive reset behavior is gated by explicit environment flag.

7. Server-Side Data Visibility and Storage Reality

7.1 What is stored

  • Accounts: hashed credentials, identity/public metadata.
  • Sessions/devices: session secrets, device IDs/names, push tokens/platforms.
  • Message operations: offline queue payload objects, unread/read state, delivery metadata.
  • Files: uploaded binary objects plus metadata and tokenized download controls.

7.2 Confidentiality boundary (important)

  • Main message bodies are encrypted client-side in normal encrypted paths.
  • Server still sees substantial metadata needed for routing and state.
  • Push/preview features can expose limited content context (short text preview class), reducing strict confidentiality claims.

7.3 Operator-readability conclusion

  • Full plaintext conversation reconstruction is not the default behavior.
  • Partial context/metadata visibility remains real and material.
  • Therefore, MiraiChat should be modeled as metadata-visible E2EE, not metadata-minimized anonymous routing.

8. Multi-Device Security Evaluation

8.1 Link flow posture

  • Device-link exchange is encrypted end-to-end between devices using ephemeral key agreement.
  • Main device authorization and local user interaction checks are present.

8.2 Risk implications

  • Linked-device onboarding transfers highly sensitive account material to bootstrap secondary device usability.
  • This is operationally effective but raises endpoint trust requirements.
  • Security guarantee remains as strong as the weakest linked endpoint.

8.3 Net effect on E2EE guarantees

  • Multi-device support does not inherently break E2EE transport semantics.
  • It does enlarge the trusted endpoint perimeter and compromise surface.

9. Abuse, Misuse, and Governance Surface

  • User block/report workflows and local block-list behavior are present.
  • Strong centralized moderation/governance controls are not a dominant architecture goal.
  • Platform behavior aligns more with secure communication tooling than with heavily moderated social content platforms.

10. Comparative Positioning (Conceptual)

Signal

Signal remains stronger in strict minimization posture and anti-operator confidentiality assumptions. MiraiChat is closer to practical privacy with greater operational flexibility (including self-hosting).

WhatsApp

Similar high-level E2EE-at-content concept, but both retain meaningful metadata planes. MiraiChat differentiates by optional self-hosting control but shifts operator burden accordingly.

Telegram

MiraiChat’s default encrypted chat path is stronger than Telegram cloud-chat defaults (where E2EE is not universal by default).

11. Residual Risk Register (Condensed)

High relevance

  • Endpoint compromise (mobile malware / rooted device / local exfiltration).
  • Weak user PIN entropy despite PBKDF2 work factor.
  • Misconfigured TLS, pin trust workflows, or admin access controls in self-hosted deployments.

Medium relevance

  • Metadata privacy leakage at routing and push layers.
  • Multi-device trust sprawl without strict endpoint hygiene policies.

Lower relevance (given normal assumptions)

  • Primitive-level cryptographic weakness (no immediate primitive red flags observed).

12. Professional Trust Assessment

A reasonably cautious organization or advanced user can trust MiraiChat for private communication where the threat model is:

  • protection from passive interception and ordinary server-side plaintext handling,
  • moderate resistance to replay/tampering,
  • controlled self-hosted or trusted-operator environment.

MiraiChat is less suitable as-is for threat models requiring:

  • near-zero metadata exposure,
  • strict anti-operator confidentiality guarantees across all message-adjacent channels,
  • high-assurance endpoint compromise resistance.

13. Deployment and Operations Guidance (No Feature Speculation)

  • Treat endpoint hardening as first-order control (MDM, OS patching, jailbreak/root detection policy, app-lock posture).
  • Enforce strong admin controls (TOTP, strict allowlist, secret rotation discipline).
  • Operate robust TLS lifecycle and monitor trust/pin state transitions.
  • Maintain retention minimization for offline queues/files consistent with regulatory and threat requirements.
  • Document user-facing expectation boundaries: encryption does not defeat compromised devices.

14. Final Professional Conclusion

MiraiChat demonstrates a serious and technically grounded secure-messaging implementation with meaningful E2EE protections and modern trust controls. It is appropriate for many practical private-conversation use cases when deployed and operated carefully.

The system should be positioned honestly as strong practical privacy with explicit operational and metadata caveats, rather than as absolute secrecy against all privileged infrastructure actors.
